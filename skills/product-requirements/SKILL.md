# Product Requirements

> Skill: product-requirements | v1.0.0 | for: pe
> 适用场景：用户需求分析、PRD 编写、需求优先级评估、需求变更管理

## 描述

对用户需求进行分析和结构化，输出规范的 PRD（产品需求文档），
评估需求可行性，为 COO 的产品决策提供支撑。

## 前置条件

- 了解 OpenClaw 生态的功能现状
- 能与 qm 协作获取质量数据

## 输入

| 字段 | 类型 | 说明 |
|------|------|------|
| user_request | string | 用户原始需求描述 |
| requestor | string | 需求提出方 |
| priority_hint | string | 优先级提示（可选）|

## 输出格式

```markdown
# 产品需求文档：{需求名称}

> 需求编号：PRD-{YYYYMMDD}-{序号}
> 需求方：{requestor}
> 产品经理：pe
> 创建日期：{YYYY-MM-DD}

## 1. 需求概述
### 用户故事
作为 {角色}，我希望 {功能}，以便 {价值}。

### 背景
{描述需求的背景和动机}

## 2. 功能需求

### 2.1 核心功能

| 功能编号 | 功能名称 | 描述 | 验收标准 |
|---------|---------|------|---------|
| F-01 | ... | ... | 可衡量的标准 |

### 2.2 用户交互流程
```text
{用户操作流程}
```

## 3. 非功能需求

| 维度 | 要求 |
|------|------|
| 性能 | 响应时间 ≤Xms |
| 可用性 | 系统可用性 ≥99% |
| 安全 | 符合 QL_SOP_002 规范 |

## 4. 需求可行性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 技术可行性 | ✅可行/⚠️有条件/❌不可行 | |
| 时间成本 | 低/中/高 | |
| 依赖关系 | ... | |

## 5. 优先级建议

| 优先级 | 理由 |
|--------|------|
| P0/P1/P2/P3 | |

## 6. 后续步骤
- [ ] 评审通过后分配给对应 Agent
- [ ] ...
```

## 示例

**输入：**
```
用户（Administrator）希望能够通过自然语言查询知识库中已有的信息，而不需要记住具体的卡片名称
提出方：Administrator
```

**输出：**
```
# 产品需求文档：自然语言知识库搜索

> 需求编号：PRD-20260627-001
> 需求方：Administrator
> 创建日期：2026-06-27

## 1. 需求概述
### 用户故事
作为 Administrator，我希望通过自然语言提问找到知识库中的相关信息，
而不需要记住具体的卡片名称。

### 背景
当前知识库需要用户知道具体卡片名称才能查找，限制了知识的可发现性。

## 2. 功能需求
| 编号 | 功能 | 描述 | 验收标准 |
|------|------|------|---------|
| F-01 | 自然语言搜索 | 支持"有哪些关于XX的卡片"这样的查询 | 搜索结果 Top3 包含相关内容 |
| F-02 | 模糊匹配 | 不记得确切名称时也能找到 | 任意关键词匹配 |

## 4. 可行性评估
| 维度 | 评估 |
|------|------|
| 技术可行性 | ⚠️ 需要引入向量数据库（Chroma） |
| 时间成本 | 中（需要架构评审） |

## 5. 优先级
**P1** — 有较高使用价值，且不影响现有功能

## 6. 后续步骤
- [ ] architecture-review 评估 Chroma 集成可行性
- [ ] pe 制定实施计划
```
